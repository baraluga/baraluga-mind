## Main findings

1. Build output can differ from the checked-out source

All three tox configurations run `ruff format` without `--check`. The package is built afterward, meaning CI can format files and publish code that was never committed.

`user_microservice` also generates `user/version.yml` during tox. This makes the release artifact intentionally different from the checkout.

This is the highest migration/reproducibility risk.

2. The entire tox suite runs twice

Each pipeline runs:

- `tox` directly on the hosted agent.
- The same `tox` suite again inside `public.ecr.aws/lambda/python:3.13`.

The client libraries declare `format`, `lint`, `py311`, and `py313`, so four environments are invoked twice. `user_microservice` declares three environments and invokes them twice.

This provides Lambda-runtime validation, but repeats formatting, linting, dependency resolution, and unit tests.

3. Feature-branch and PR validation runs are duplicated

Every branch matches `trigger: '*'`, while `master` also has a blocking build-validation policy. Run history shows paired runs for the source branch and PR merge ref.

For example, a `user_microservice` PR repeatedly produced both:

- `refs/heads/PR-14919`
- `refs/pull/8160/merge`

This approximately doubles validation consumption for active PRs.

4. Three near-identical pipelines are maintained independently

The repositories contain 189 total pipeline lines plus identical copies of `test_aws_lambda.sh` and `.azure/aws.sh`.

The material differences are only:

- Coverage is published for `user_microservice`, but not the other two.
- `user_microservice` uploads `dist/*`.
- The other two upload only `dist/*.gz`, discarding built wheels.
- Matrix display names differ, sometimes incorrectly.

A shared parameterized template would eliminate most of this drift.

5. CI and package release are coupled

The same job tests, builds, authenticates, and publishes. Publishing is correctly restricted to `master`, but there is no immutable artifact passed from CI to a separate release job/stage.

Consequences include:

- Retrying a successful publication may fail because the package version already exists.
- Release credentials and permissions are present in the general CI job.
- There is no independent approval or release audit boundary.

6. Reproducibility is weak

Every run dynamically installs unpinned versions of:

- `uv`
- `twine`
- `build[uv]`
- `tox-uv`

The Lambda image and `ubuntu-24.04` hosted image are also floating references. A pipeline can therefore change behavior without a repository commit.

7. Unnecessary permission and secret coupling

All three definitions use `jobAuthorizationScope: projectCollection`, which is broader than the project-scoped access these builds appear to need.

They also import `DFFVariableGroup`, whose only variable is `sentry-auth-token`; it is not referenced by these pipelines. Removing it would reduce secret exposure and legacy DFF coupling.

The pipeline also creates `UV_EXTRA_INDEX_URL` through a logging command without marking the resulting variable secret.

8. Configuration drift and misleading names

`user_microservice` and `web_common` have a matrix entry named `Python311` whose actual value is Python 3.13. A one-entry matrix adds indirection without providing a real compatibility matrix.

Coverage XML is generated in every repository, but only `user_microservice` publishes it.

## Recommended target

Prioritized changes:

1. Make formatting check-only: `ruff format --check`.
2. Generate version metadata in an explicit build step and publish an immutable artifact.
3. Separate CI validation from master package publication.
4. Run feature branches through either CI triggers or PR validation, not both.
5. Put the common flow in one versioned Azure Pipelines template.
6. Split host checks from Lambda compatibility testing so the entire suite is not repeated.
7. Pin tool versions and preferably the Lambda image digest.
8. Remove the unused variable group and `.azure/aws.sh`; restrict job authorization to project scope.
9. Standardize coverage publication and decide explicitly whether wheels should be published.

Overall: these are straightforward pipelines to migrate structurally, but I would treat `user_microservice` as the pilot because its generated version file, larger test surface, coverage step, and 25% retained-run failure rate expose the important edge cases. Its latest `master` run is currently successful; the sampled failures occurred in the unit-test task on a PR rather than in Azure orchestration.

Pushing to a branch with an open PR creates two independent events:

- `push`: tests the branch’s head commit.
- `pull_request` (`synchronize`): normally tests GitHub’s synthetic merge of that branch into the PR target.

GitHub does not deduplicate them. So they’re not technically identical, but often duplicate most work. [GitHub documents the event and SHA differences here.](https://docs.github.com/en/actions/reference/workflows-and-actions/events-that-trigger-workflows#pull_request)

```yaml
on:
  pull_request:
    branches: [main]

  push:
    branches: [main]
```

That gives you one PR run while developing, then one `push` run after merging to `main`. Publishing should generally be guarded to `main` or tags as well.

Also add concurrency to cancel outdated runs after subsequent pushes:

```yaml
concurrency:

```

Concurrency saves work from superseded commits, though the branch filtering is what removes the push/PR double run. [GitHub concurrency documentation.](https://docs.github.com/en/actions/how-tos/write-workflows/choose-when-workflows-run/control-workflow-concurrency)

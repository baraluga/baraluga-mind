      PARTITION BY interconnector_name, interval_start_jst

  FROM actual_flow

## Main pain points encountered

1. Repository transfer method and permissions

The Git history itself is relatively easy. The complications were:

- Different migration options preserve different things.
- Full mirror pushes can include ADO-specific hidden refs that GitHub rejects.
- Preserving PR history requires additional tooling and migrator permissions; we decided it wasn’t worth the added access and complexity.
- Native organization transfers can wait on owner approval and temporarily make the repository unavailable.
- Developers continuing to push during migration creates divergence and leftover commits.

For our needs, branches, tags, commit history, and commit messages were the durable record. PR history was optional. See the [recorded decision](/Users/qn5792/baraluga-mind/wiki/decisions/2026-07-09-skip-pr-history-for-walnut-migration.md) and [technical-team discussion](/Users/qn5792/baraluga-mind/sources/meetings/2026-07-09-1515-granola-technical-team-standup.md).

2. Pipelines do not migrate with the repository

This was probably the largest effort.

ADO pipelines contain platform-specific concepts:

- ADO tasks and templates
- variable groups and secrets
- service connections
- agent assumptions
- approval and deployment stages
- Azure Artifact authentication
- branch-based triggers

These need to be mapped or rewritten as GitHub Actions. A workflow turning green at checkout doesn’t prove that the application can build, test, publish, or deploy.

The migration planning notes explicitly identified pipeline recreation—not repo transfer—as most of the work: [Walnut Migration Planning](/Users/qn5792/baraluga-mind/sources/meetings/2026-07-08-1330-granola-walnut-migration-planning.md).

3. Runner capability and network trust

“Does the Walnut Ubuntu runner work?” turned out to be too broad a question. It could run GitHub Actions, but the meaningful question was:

> Can the runner access, authenticate to, and trust every private dependency and deployment endpoint used by this project?

We encountered:

- private network dependencies
- Artifactory authentication
- corporate CA certificates not trusted by the runner
- `UnknownIssuer` and self-signed certificate-chain errors
- differences between local developer machines and clean CI runners

A local build can work because the managed laptop already trusts ENGIE certificates. The same command can fail on a runner until the certificate chain is installed.

4. Artifact migration is a separate workstream

ADO gave us repos, pipelines, and artifacts in one platform. Moving the Git repo doesn’t move the package ecosystem.

Questions that had to be answered:

- Where will each Python, npm, Docker, or shared package live?
- Which repository is used for publishing?
- Which repository is used by consumers?
- Which environments are required: dev, prod, expose, scratch?
- How will CI authenticate?
- Which certificates and network routes are required?
- Can the old ADO feed still receive new versions after cutover?

For SMP, `smp-common` made this concrete: direct installation from a private GitHub repo would require every consumer to have repository access and a PAT. Publishing through Artifactory is the more scalable model. The internal [Artifactory with GitHub guide](https://tractebel-et.atlassian.net/wiki/spaces/DED/pages/5531664386/Artifactory+with+Github) documents this path.

Pipeline migration exposed credentials that were implicit or centrally managed in ADO:

- package registry credentials
- AWS deployment identities
- GitHub repository access
- Airflow `git-sync` credentials
- environment-specific secrets
- organization-level versus repository-level secrets

The migration cannot be called complete until the target pipeline proves that these work in the real environment—not merely in a test script.

6. Governance was not ready by default

Creating a GitHub repository doesn’t reproduce the old governance model.

We still needed to define:

- repository visibility
- owners and teams
- branch protection and rulesets
- required checks
- reviewer requirements
- environment approvals
- secret ownership
- who has administrative access

This should be treated as part of the migration template, not configured ad hoc after repositories arrive.

Some repositories were runtime dependencies. For example, Airflow continuously fetched DAG code through `git-sync`. Changing or transferring that repository without coordinating its consumers could break a running environment.

The safe sequence was:

1. Create and populate the target.
2. Rebuild CI/CD.
3. Validate access and runtime behavior.
4. Perform a final synchronization.
5. Update downstream consumers.
6. Make the old source read-only.
7. Archive it only after the target is proven.

## Best practices

### Before migrating

- Inventory more than repositories. Include pipelines, artifacts, secrets, service connections, runners, deployment targets, webhooks, branch policies, and downstream consumers.
- Classify repositories by complexity: source-only, simple CI, deployment pipeline, shared package, and runtime-critical.
- Decide what history is genuinely required. Don’t pursue PR-history preservation automatically.
- Assign one owner for the application migration and named contacts for Walnut, infrastructure, artifacts, and security.
- Confirm the target repository visibility and governance before copying anything.
- Pilot with one representative repository—not necessarily the easiest one.

### Repository migration

- Prefer a staged private mirror where possible.
- Preserve branches, tags, and commit history explicitly.
- Avoid broad mirror pushes when the source contains platform-specific hidden refs.
- Copy repository metadata too: description, default branch, topics, owners, and visibility.
- Keep the source operational during validation.
- Schedule a short write freeze or perform a final incremental sync before cutover.
- Avoid approval-dependent native transfers for runtime-critical repositories unless the receiving owner is ready.

### Pipeline migration

- Treat it as a rewrite and validation exercise, not file conversion.
- Keep workflows in the repository as code.
- Map every old pipeline responsibility to a GitHub equivalent.
- Validate progressively:

  1. Checkout
  2. Dependency installation
  3. Tests and quality gates
  4. Artifact publication
  5. Authentication to the target environment
  6. Deployment
  7. Rollback or recovery path

- A green dependency-install job is not proof that full CI/CD works.
- Create reusable workflows or actions for common organization concerns such as certificate installation, authentication, and standard quality checks.

### Artifact migration

- Inventory package producers and consumers separately.
- Establish the Artifactory repositories before cutting pipelines over.
- Test publishing and consuming using an unofficial/test version.
- Standardize package index names and credential contracts.
- Test from the actual Walnut runner.
- Diagnose failures by layer: DNS/network → TLS trust → authentication → authorization → package resolution.
- Do not keep publishing new releases to the old ADO feed after consumers have moved.

### Cutover

- Use explicit acceptance criteria per repository.
- Prove the target pipeline and deployment before disabling the source.
- Update runtime references, documentation, badges, package URLs, and developer clone instructions.
- Make the source read-only first; archive later.
- Keep a migration record containing source, target, owner, pipeline status, artifact status, cutover date, and remaining blockers.
